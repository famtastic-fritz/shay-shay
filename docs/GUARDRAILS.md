# Shay Guardrails

Non-negotiable build rules for all contributors and agents. These rules apply to every phase and every commit.

## 1. Additive-Only Development

- Never delete, rename, or mutate files outside `/Users/famtasticfritz/famtastic/shay-shay-build/` without an explicit phase unlock instruction.
- All work happens inside the build directory.
- The live Python gateway at `/Users/famtasticfritz/famtastic/shay-shay/` is reference material only until Phase 9.
- Do not copy code from the live gateway into the build tree; instead, read the Python implementation, understand the semantics, and rewrite in TypeScript.
- Once a file is written in the build tree, treat it as owned — do not rewrite it from scratch without explicit approval. Patch or extend it instead.

## 2. Live Gateway is Read-Only

- The Python gateway in `shay-shay/` is the source of truth for the existing Agent Spine.
- Use it to understand:
  - The three-phase core (Config, Event Router, Credential Vault)
  - The Doctor's gate signature and verification flow
  - Integration patterns between subsystems
- Do not modify, patch, or refactor anything in `shay-shay/` until Phase 9 ships.
- If a bug is found in the live gateway, document it in a known-gaps note and plan a Phase 10 fix; do not hotfix.

## 3. Zero AI References

Strict, zero-tolerance enforcement:

- No mention of "Claude", "Haiku", "Sonnet", "Opus", "Gemini", or any LLM name in:
  - Source code (TypeScript, JavaScript, JSON, YAML)
  - Code comments and docstrings
  - Commit messages and commit bodies
  - Documentation (markdown, text)
  - Configuration files (tsconfig.json, package.json, etc.)
  
- No use of words like "generated", "assisted", "AI-powered", "co-authored", or "created by" referring to tools or automation.
- This is enforced by the gate (runGate checks source).

## 4. Conventional Commits

Every commit must follow conventional commit format:

```
<type>(<scope>): <subject>

<body (optional)>

<footer (optional)>
```

Valid types: `feat`, `fix`, `chore`, `docs`, `test`, `refactor`, `perf`.

Examples:
```
feat(core): add config registry and schema validation
fix(doctor): correct gate verdict assignment
docs: update phase workflow template
```

- No emojis in commit messages.
- No AI references anywhere (see Rule 3).
- Scope is optional but recommended.
- Commit messages should be imperative and present tense ("add", "fix", not "added", "fixed").

## 5. Gate Before Commit

The Doctor's gate must pass before any commit is created:

```bash
cd /Users/famtasticfritz/famtastic/shay-shay-build
node -e 'import("@shay/doctor").then(m => m.runGate(".")).then(result => { console.log(JSON.stringify(result, null, 2)); process.exit(result.pass ? 0 : 1); })'
```

Expected output on pass:
```json
{
  "pass": true,
  "subsystems": { ... },
  "timestamp": "..."
}
```

If `pass` is `false`, fix all issues before committing.

## 6. One Phase at a Time

- Do not begin Phase N+1 until Phase N's gate passes and the commit is made.
- Each phase has a clear input state (what exists) and output state (what must exist).
- Phases may be split across multiple sessions, but all work within a phase is additive and gates must pass before moving on.
- Decompose each phase into file groups using the PHASE-WORKFLOW-TEMPLATE.md pattern.

## 7. File Ownership and Collision Domains

- Each file is owned by the group that creates it.
- Collision domains are: package manifest files, config files, schema files, and interface files.
- If two groups create the same file, clearly identify the conflict in the phase plan before building.
- Once created, a file is owned — patches to it are welcome, but rewrites require review.

## 8. TypeScript and Type Safety

- All TypeScript must type-check with `tsc --noEmit` (no transpilation required, we emit to JavaScript).
- Interfaces are minimal and used sparingly; focus on clarity and stability.
- Use JSON Schema for domain models and validation logic (stored in `schemas/<domain>.schema.json`).
- Do not mix TypeScript interfaces with JSON Schema; choose one per domain.

## 9. Package Structure

All work is under `packages/<name>/`:

```
packages/<name>/
├── src/
│   ├── index.ts         (entry point)
│   ├── types.ts         (if needed)
│   └── ...
├── dist/
│   └── index.js         (generated by build)
├── package.json
└── tsconfig.json
```

- Each package is published as `@shay/<name>`.
- No circular dependencies between packages.

## 10. Configuration and Schema Files

- Global config is `shay.config.yaml` at repo root.
- Domain schemas are in `schemas/<domain>.schema.json`.
- Package-specific config is in `packages/<name>/package.json` only.
- All JSON files must be valid JSON.

## How Violations Are Caught

The Doctor's gate checks:

1. **File inventory:** All expected files for the phase exist and have content.
2. **TypeScript coherence:** `tsc --noEmit` passes for all packages.
3. **AI references:** Scans all source files, comments, and key docs for forbidden terms.
4. **Conventional commit format:** (checked at commit time if integrated with git hooks).

Violations block the gate and prevent commit.
