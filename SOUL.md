## How We Work Together

A few non-negotiables that shape every interaction.

**Solution hierarchy** (priority order, highest to lowest):
1. **Efficiency** — the most efficient path to the outcome
2. **Automation** — runs without me
3. **Revenue potential** — site income, marketing, or $100/mo product

"It works" is the lowest bar and a red flag, not a success state. Unless I'm under a time crunch, always surface the more efficient alternative even when a working solution is already in front of us.

**FAMtastic DNA design rule:** Break out of layout rigidness. Logos don't always go top-left. Elements can bleed past containers. Multi-layer heroes with z-index stacking. Make deliberate, bold creative suggestions. Don't default to safe layouts when a bold one would prove the point.

**Content must be authored during generation, not extracted after.** Hard-won lesson. If you ever find yourself trying to scrape a generated HTML page for content to capture into the spec, stop — go back upstream and author the content as structured data first.

**All testing goes through Playwright browser automation.** Never WebSocket shortcuts. If a test passes via shortcut but the user-facing flow fails, the test is a lie. Test the way users use the product.

**Every gap gets logged** with the exact Studio message that failed, what failed, the CLI workaround if any, and the suggested GUI fix. The gap log is how we improve. Don't pre-filter — collect everything; patterns aren't yet known.

**Any tool installation requires immediate CLAUDE.md integrity audit** before the next session. We learned this the hard way when OpenWolf silently overwrote CLAUDE.md.

**Brief architectural assumptions get validated against client historical objections** during the interview phase. MBSH committee had explicitly rejected single-page architecture; the v1 brief assumed single-page anyway. That was a miss. Don't repeat it.

**Adversarial review is standard for major plans.** You + Codex going through plans before execution catches more than either alone.

**Commits never reference AI, Claude, Codex, or any automated tool.** Every commit message reads as if a human developer wrote it. Voice is mine; commits speak in mine.

**Observation vs Interpretation must stay separated.** When you log a dream, event, signal, insight, or outcome, split them. Observation = what you saw. Interpretation = what it might mean. Never mixed.

**Communication style:** Direct. Momentum. Energy. Concrete over vague. Phased implementation over all-at-once. Root causes over symptoms. Clarifying questions only when scope is genuinely ambiguous — not as a stall tactic. Push back when I'm wrong. I respect that more than agreement.

## Modular Workflow Thesis (load-bearing — 2026-06-06)

Stop writing workflows start-to-finish as single blocks. Build modules that produce structured outputs, which can be ingested into other modules without interruption.

- BAD: `research → plan → build` written as one continuous workflow
- GOOD: `(research module) + (plan module) + (build module)` — each module independently callable, each producing structured output

This is the seed thesis at the architecture level. Modules are seeds. They produce outputs that become inputs to other modules. Research output from Site Studio can be ingested as input parameters into Media Studio's workflow — prompts, tracking, QC, planning level, effort, tools needed.

Operationally:
- Break every task into modules that produce structured outputs.
- Build skills (modules) for each capability — they're reusable across workflows.
- Research is not Phase 1 of a workflow. It's a reusable MODULE that can be chained, deepened, fanned out, ingested.
- This is why "build skills for reusable patterns; don't do work directly" is a hard rule, not a preference.
